# Knowledge Base Guidelines

This section contains documentation about the knowledge base itself - how to create, maintain, and improve our documentation.

## Purpose

The guidelines folder is where we document our documentation practices. This ensures:

- **Consistency**: All technology docs follow the same high standards
- **Quality**: Clear guidelines for what makes great documentation
- **Maintainability**: Processes for keeping docs up to date
- **Growth**: Continuous improvement of our knowledge base

## Contents

### [Creation Guide](./creation-guide.md)

Comprehensive guide for creating new technology documentation. Includes:

- Research methodology (4-phase approach)

**Use this guide when**: You need to document a new technology or library in the codebase.

## Quick Reference

### Creating New Technology Documentation

1. Read the [Creation Guide](./creation-guide.md)
2. Use the template structure (Phase 1-4, ~3-4 hours)
3. Follow quality standards checklist
4. Get peer review
5. Update [knowledge-base-updates.md](../../ideas/knowledge-base-updates.md)

### Updating Existing Documentation

1. Check quarterly review schedule
2. Verify version numbers and links
3. Test all code examples
4. Update based on team feedback
5. Document changes in the file's history section

### Documentation Standards

- **Minimum**: 400+ lines, official links, 3+ examples, troubleshooting
- **Excellence**: Real codebase examples, performance notes, comparisons
- **Maintenance**: Review quarterly or on major version releases

## Continuous Improvement

This guidelines documentation should evolve as we learn:

- What makes our docs most useful
- Where developers struggle to find information
- How to make docs more searchable and scannable
- Better ways to present complex information

**Feedback Welcome**: If you have ideas for improving our documentation practices, add them to the creation guide or propose new guideline documents.
